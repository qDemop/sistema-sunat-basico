BEGIN;

CREATE OR REPLACE FUNCTION admin.fn_periodo_valido(p_periodo TEXT)
RETURNS BOOLEAN
LANGUAGE sql
IMMUTABLE
AS $$
    SELECT p_periodo ~ '^[0-9]{4}-(0[1-9]|1[0-2])$';
$$;

CREATE OR REPLACE FUNCTION payroll.fn_redondear_monto(p_monto NUMERIC)
RETURNS NUMERIC(12,2)
LANGUAGE sql
IMMUTABLE
AS $$
    SELECT round(COALESCE(p_monto, 0), 2)::NUMERIC(12,2);
$$;

CREATE OR REPLACE FUNCTION payroll.fn_calcular_afp(p_salario_bruto NUMERIC)
RETURNS NUMERIC(12,2)
LANGUAGE sql
IMMUTABLE
AS $$
    SELECT payroll.fn_redondear_monto(COALESCE(p_salario_bruto, 0) * 0.10);
$$;

CREATE OR REPLACE FUNCTION payroll.fn_calcular_onp(p_salario_bruto NUMERIC)
RETURNS NUMERIC(12,2)
LANGUAGE sql
IMMUTABLE
AS $$
    SELECT payroll.fn_redondear_monto(COALESCE(p_salario_bruto, 0) * 0.13);
$$;

CREATE OR REPLACE FUNCTION payroll.fn_calcular_gratificacion(p_salario_bruto NUMERIC)
RETURNS NUMERIC(12,2)
LANGUAGE sql
IMMUTABLE
AS $$
    SELECT payroll.fn_redondear_monto(COALESCE(p_salario_bruto, 0) / 6);
$$;

CREATE OR REPLACE FUNCTION payroll.fn_calcular_cts(
    p_salario_bruto NUMERIC,
    p_gratificacion NUMERIC
)
RETURNS NUMERIC(12,2)
LANGUAGE sql
IMMUTABLE
AS $$
    SELECT payroll.fn_redondear_monto((COALESCE(p_salario_bruto, 0) + COALESCE(p_gratificacion, 0)) / 12);
$$;

CREATE OR REPLACE FUNCTION payroll.fn_calcular_horas_extra(
    p_salario_base NUMERIC,
    p_horas_primeras_dos NUMERIC,
    p_horas_posteriores NUMERIC
)
RETURNS NUMERIC(12,2)
LANGUAGE sql
IMMUTABLE
AS $$
    SELECT payroll.fn_redondear_monto(
        ((COALESCE(p_salario_base, 0) / 240) * 1.25 * COALESCE(p_horas_primeras_dos, 0)) +
        ((COALESCE(p_salario_base, 0) / 240) * 1.35 * COALESCE(p_horas_posteriores, 0))
    );
$$;

CREATE OR REPLACE FUNCTION accounting.fn_tasa_igv_activa(p_fecha DATE)
RETURNS NUMERIC(7,4)
LANGUAGE sql
STABLE
AS $$
    SELECT COALESCE(
        (
            SELECT ct.porcentaje
            FROM admin.config_tributaria ct
            WHERE ct.codigo = 'IGV'
              AND ct.activo = TRUE
              AND ct.fecha_inicio <= p_fecha
              AND (ct.fecha_fin IS NULL OR ct.fecha_fin >= p_fecha)
            ORDER BY ct.fecha_inicio DESC
            LIMIT 1
        ),
        18
    )::NUMERIC(7,4);
$$;

CREATE OR REPLACE FUNCTION accounting.fn_calcular_igv(
    p_base_imponible NUMERIC,
    p_tipo_operacion TEXT,
    p_tasa NUMERIC DEFAULT 18
)
RETURNS NUMERIC(12,2)
LANGUAGE sql
IMMUTABLE
AS $$
    SELECT CASE
        WHEN p_tipo_operacion = 'Gravada'
            THEN round(COALESCE(p_base_imponible, 0) * (COALESCE(p_tasa, 18) / 100), 2)::NUMERIC(12,2)
        ELSE 0::NUMERIC(12,2)
    END;
$$;

CREATE OR REPLACE FUNCTION accounting.fn_calcular_total(
    p_base_imponible NUMERIC,
    p_base_exonerada NUMERIC,
    p_igv NUMERIC
)
RETURNS NUMERIC(12,2)
LANGUAGE sql
IMMUTABLE
AS $$
    SELECT round(
        COALESCE(p_base_imponible, 0) +
        COALESCE(p_base_exonerada, 0) +
        COALESCE(p_igv, 0),
        2
    )::NUMERIC(12,2);
$$;

CREATE OR REPLACE FUNCTION audit.fn_registrar_evento(
    p_id_usuario BIGINT,

    p_modulo VARCHAR,
    p_entidad VARCHAR,
    p_entidad_id VARCHAR,
    p_operacion VARCHAR,

    p_detalle JSONB DEFAULT NULL,

)
RETURNS BIGINT
LANGUAGE plpgsql
AS $$

    v_id_audit BIGINT;

    INSERT INTO audit.audit_log (
        id_usuario,
        username,
        modulo,
        entidad,
        entidad_id,
        operacion,
        resultado,
        detalle,
        correlation_id
    )
    VALUES (
        p_id_usuario,
        p_username,
        p_modulo,
        p_entidad,
        p_entidad_id,
        p_operacion,
        p_resultado,
        p_detalle,
        p_correlation_id
    )
    RETURNING id_audit INTO v_id_audit;

    RETURN v_id_audit;
END;
$$;

# Database Normalization Specification

Source: `Entregable.pdf`, pages 78-84.

## Initial Unnormalized Structure

The source document starts from a conceptual unnormalized table named `PLANILLA_GENERAL` that mixes:

- Payroll data.
- Employee data.
- Voucher data.
- User and role data.
- Multivalued discounts such as AFP and CTS in one field.

## Problems Identified

- Employee data repeats for every voucher.
- Discount values are stored as multivalued text.
- Entities are not separated.
- Salary changes would require multiple updates.
- Deleting a voucher could accidentally remove employee information.

## First Normal Form

Rules:

- All attributes must be atomic.
- No repeated groups.
- Each row must be unique.

Applied changes:

- Split discounts into atomic columns: AFP, ONP, CTS, gratification.
- Separate employee, voucher, and user data conceptually.
- Use a composite key during transition: `(id_empleado, periodo, id_comprobante)`.

Remaining issue:

- Partial dependencies still exist because employee fields depend only on `id_empleado`, voucher fields depend only on `id_comprobante`, and user fields depend only on `id_usuario`.

## Second Normal Form

Rules:

- Must be in 1NF.
- Non-key attributes must depend on the entire key.

Applied changes:

- Create `EMPLEADO`.
- Create `PLANILLA`.
- Create `COMPROBANTE`.
- Create `USUARIO`.

Remaining issue:

- Transitive dependencies remain, such as `tipo_descuento` determining a discount percentage and `rol_usuario` determining access level.

## Third Normal Form

Rules:

- Must be in 2NF.
- Non-key attributes must depend directly on the primary key.

Applied changes:

- Create `TIPO_DESCUENTO`.
- Update `EMPLEADO` to reference `TIPO_DESCUENTO`.
- Create `ROL`.
- Update `USUARIO` to reference `ROL`.
- Create `TIPO_COMPROBANTE`.
- Update `COMPROBANTE` to reference `TIPO_COMPROBANTE`.
- Create `LIBRO_CONTABLE`.
- Create `DETALLE_PLANILLA`.

## BCNF Target

Rule:

- For every non-trivial functional dependency `X -> Y`, `X` must be a superkey.

The final schema is considered BCNF because each non-key attribute depends on the table key and no non-key attribute determines another non-key attribute.

## BCNF Benefits

- Removes redundancy.
- Prevents update, insert, and delete anomalies.
- Enforces referential integrity.
- Improves JOIN efficiency with indexes.
- Supports future modules without redesigning existing tables.

# Stored Procedure Specifications

This document specifies stored procedures only. It does not provide executable SQL.

## sp_calcular_planilla(periodo)

### Purpose

Calculate or recalculate monthly payroll for all active employees for a selected period.

### Inputs

| Name | Type | Rule |
|---|---|---|
| periodo | string | Required, format `YYYY-MM`. |

### Preconditions

- Period is valid.
- At least one active employee exists.
- Each active employee has salary base, department, and discount type.
- Discount type exists and is active.
- Existing payroll rows for the period are either absent or have status `Draft`.
- Existing payroll rows with status `Finalized` block recalculation.

### Processing Rules

For each active employee:

1. Calculate hourly rate from base salary using 240 monthly hours.
2. Load approved `HORAS_EXTRA` records for the employee and period.
3. Calculate overtime amount using hourly rate, overtime hours, and surcharge percentage.
4. Calculate gross salary as base salary plus overtime amount.
5. Apply AFP 10% when discount type is AFP.
6. Apply ONP 13% when discount type is ONP.
7. Calculate gratification as `gross_salary / 6`.
8. Calculate CTS monthly estimate as `(gross_salary + gratification) / 12`.
9. Calculate total discounts.
10. Calculate net salary as `gross_salary - total_discounts`.
11. Insert missing `PLANILLA` rows with status `Draft`, or update existing `Draft` rows.
12. Replace detail rows in `DETALLE_PLANILLA` for the updated draft payroll rows.

### Outputs

Per employee result:

- id_empleado.
- id_departamento.
- nombre.
- salario_bruto.
- horas_extra_total.
- afp.
- onp.
- cts.
- gratificacion.
- descuentos.
- neto.
- estado.

### Transaction Rule

The whole period calculation must be atomic. If one employee calculation fails, the transaction must roll back without partial payroll updates.

### Performance Requirement

Must complete for 100 employees in 30 seconds or less.

## sp_finalizar_planilla(periodo)

### Purpose

Finalize all draft payroll rows for a selected period and prevent further recalculation.

### Inputs

| Name | Type | Rule |
|---|---|---|
| periodo | string | Required, format `YYYY-MM`. |

### Preconditions

- Period is valid.
- Draft payroll rows exist for the period.
- No payroll row for the period has validation errors.

### Processing Rules

1. Validate that all active employees have a payroll row for the period.
2. Change `PLANILLA.estado` from `Draft` to `Finalized`.
3. Set `fecha_finalizacion`.
4. Write audit event for finalization.

### Outputs

- periodo.
- total_empleados.
- total_bruto.
- total_descuentos.
- total_neto.
- estado.

## sp_generar_libro(tipo, periodo)

### Purpose

Generate a new version of the SUNAT purchase or sales book for a selected period.

### Inputs

| Name | Type | Rule |
|---|---|---|
| tipo | string | Required, expected `compras` or `ventas`. |
| periodo | string | Required, format `YYYY-MM`. |

### Preconditions

- Period is valid.
- Voucher types and book types are configured.
- Active tax configuration exists for the period.
- Vouchers are in `Registrado` state.
- `tipo = compras` maps to `COMPROBANTE.tipo_movimiento = Compra`.
- `tipo = ventas` maps to `COMPROBANTE.tipo_movimiento = Venta`.

### Processing Rules

1. Query vouchers for the selected period and movement type.
2. Include SUNAT fields: period, document identity, voucher type, series, number, date, taxable base, exempt base, IGV, total.
3. Group by voucher type.
4. Calculate base, IGV, and total by group.
5. Calculate general totals.
6. Create a new `LIBRO_CONTABLE` row with the next version for the period and book type.
7. Insert `COMPROBANTE_LIBRO` rows for each included voucher with snapshot totals and row order.
8. Preserve prior book versions without mutation.

### Outputs

- Book header.
- Book version.
- Voucher rows.
- Totals by voucher type.
- General totals.

### Performance Requirement

Must generate books for 1,000 vouchers in 60 seconds or less.

## sp_postear_asiento(id_asiento_contable)

### Purpose

Post a draft journal entry after validating accounting period, account status, and debit/credit balance.

### Inputs

| Name | Type | Rule |
|---|---|---|
| id_asiento_contable | int | Required. |

### Preconditions

- Journal entry exists with status `Draft`.
- Accounting period is `Open`.
- Journal entry has at least two detail lines.
- All referenced accounts are active.
- Total debit equals total credit.

### Processing Rules

1. Lock the journal entry and its detail rows for posting.
2. Validate the accounting period status.
3. Validate account status for every line.
4. Validate balanced debit and credit totals.
5. Change entry status to `Posted`.
6. Set `fecha_posteo`.
7. Write audit event for posting.

### Outputs

- id_asiento_contable.
- estado.
- total_debe.
- total_haber.
- fecha_posteo.

BEGIN;

CREATE OR REPLACE PROCEDURE payroll.sp_calcular_planilla(
    IN p_periodo VARCHAR(10)
)
LANGUAGE plpgsql
AS $$
DECLARE
    v_empleado RECORD;
    v_planilla_id BIGINT;
    v_estado_actual VARCHAR(20);
    v_horas_primeras_dos NUMERIC(6,2);
    v_horas_posteriores NUMERIC(6,2);
    v_horas_extra_monto NUMERIC(12,2);
    v_salario_bruto NUMERIC(12,2);
    v_afp NUMERIC(12,2);
    v_onp NUMERIC(12,2);
    v_gratificacion NUMERIC(12,2);
    v_cts NUMERIC(12,2);
    v_descuentos NUMERIC(12,2);
    v_neto NUMERIC(12,2);
    v_empleados_calculados INTEGER := 0;
BEGIN
    IF NOT admin.fn_periodo_valido(p_periodo) THEN
        RAISE EXCEPTION 'Periodo invalido: %', p_periodo;
    END IF;

    IF NOT EXISTS (SELECT 1 FROM payroll.empleado WHERE activo = TRUE) THEN
        RAISE EXCEPTION 'No existen empleados activos para calcular planilla';
    END IF;

    FOR v_empleado IN
        SELECT
            e.id_empleado,
            e.salario_base,
            td.nombre AS tipo_descuento
        FROM payroll.empleado e
        INNER JOIN payroll.tipo_descuento td ON td.id_tipo = e.id_tipo_descuento
        WHERE e.activo = TRUE
          AND td.activo = TRUE
        ORDER BY e.id_empleado
    LOOP
        SELECT p.id_planilla, p.estado
        INTO v_planilla_id, v_estado_actual
        FROM payroll.planilla p
        WHERE p.id_empleado = v_empleado.id_empleado
          AND p.periodo = p_periodo;

        IF v_estado_actual = 'Finalizada' THEN
            RAISE EXCEPTION 'La planilla del empleado % para el periodo % ya esta finalizada',
                v_empleado.id_empleado,
                p_periodo;
        END IF;

        SELECT
            COALESCE(he.horas_primeras_dos, 0),
            COALESCE(he.horas_posteriores, 0)
        INTO v_horas_primeras_dos, v_horas_posteriores
        FROM payroll.hora_extra he
        WHERE he.id_empleado = v_empleado.id_empleado
          AND he.periodo = p_periodo;

        v_horas_primeras_dos := COALESCE(v_horas_primeras_dos, 0);
        v_horas_posteriores := COALESCE(v_horas_posteriores, 0);
        v_horas_extra_monto := payroll.fn_calcular_horas_extra(
            v_empleado.salario_base,
            v_horas_primeras_dos,
            v_horas_posteriores
        );

        v_salario_bruto := payroll.fn_redondear_monto(v_empleado.salario_base + v_horas_extra_monto);

        IF v_empleado.tipo_descuento = 'AFP' THEN
            v_afp := payroll.fn_calcular_afp(v_salario_bruto);
            v_onp := 0;
        ELSIF v_empleado.tipo_descuento = 'ONP' THEN
            v_afp := 0;
            v_onp := payroll.fn_calcular_onp(v_salario_bruto);
        ELSE
            RAISE EXCEPTION 'Tipo de descuento no soportado para empleado %: %',
                v_empleado.id_empleado,
                v_empleado.tipo_descuento;
        END IF;

        v_gratificacion := payroll.fn_calcular_gratificacion(v_salario_bruto);
        v_cts := payroll.fn_calcular_cts(v_salario_bruto, v_gratificacion);
        v_descuentos := payroll.fn_redondear_monto(v_afp + v_onp);
        v_neto := payroll.fn_redondear_monto(v_salario_bruto - v_descuentos);

        IF v_planilla_id IS NULL THEN
            INSERT INTO payroll.planilla (
                id_empleado,
                periodo,
                total_bruto,
                total_descuentos,
                total_neto,
                estado
            )
            VALUES (
                v_empleado.id_empleado,
                p_periodo,
                v_salario_bruto,
                v_descuentos,
                v_neto,
                'Calculada'
            )
            RETURNING id_planilla INTO v_planilla_id;
        ELSE
            UPDATE payroll.planilla
            SET total_bruto = v_salario_bruto,
                total_descuentos = v_descuentos,
                total_neto = v_neto,
                estado = 'Calculada',
                fecha_actualizacion = now()
            WHERE id_planilla = v_planilla_id;

            DELETE FROM payroll.detalle_planilla
            WHERE id_planilla = v_planilla_id;
        END IF;

        INSERT INTO payroll.detalle_planilla (
            id_planilla,
            afp,
            onp,
            cts,
            gratificacion,
            horas_extra_monto,
            descuentos_adicionales
        )
        VALUES (
            v_planilla_id,
            v_afp,
            v_onp,
            v_cts,
            v_gratificacion,
            v_horas_extra_monto,
            0
        );

        v_empleados_calculados := v_empleados_calculados + 1;
    END LOOP;

    PERFORM audit.fn_registrar_evento(
        NULL,
        NULL,
        'Payroll',
        'payroll.planilla',
        p_periodo,
        'CALCULAR_PLANILLA',
        'Exitoso',
        jsonb_build_object('periodo', p_periodo, 'empleados_calculados', v_empleados_calculados)
    );
END;
$$;

CREATE OR REPLACE PROCEDURE accounting.sp_generar_libro(
    IN p_tipo VARCHAR(20),
    IN p_periodo VARCHAR(10)
)
LANGUAGE plpgsql
AS $$
DECLARE
    v_tipo_normalizado VARCHAR(20);
    v_tipo_movimiento VARCHAR(10);
    v_id_tipo_libro BIGINT;
    v_id_libro BIGINT;
    v_version INTEGER;
    v_fecha_inicio DATE;
    v_fecha_fin DATE;
    v_total_base NUMERIC(12,2);
    v_total_igv NUMERIC(12,2);
    v_total_general NUMERIC(12,2);
    v_comprobantes INTEGER;
BEGIN
    IF NOT admin.fn_periodo_valido(p_periodo) THEN
        RAISE EXCEPTION 'Periodo invalido: %', p_periodo;
    END IF;

    v_tipo_normalizado := lower(trim(p_tipo));

    IF v_tipo_normalizado IN ('compras', 'compra') THEN
        v_tipo_movimiento := 'Compra';
    ELSIF v_tipo_normalizado IN ('ventas', 'venta') THEN
        v_tipo_movimiento := 'Venta';
    ELSE
        RAISE EXCEPTION 'Tipo de libro no soportado: %', p_tipo;
    END IF;

    SELECT tl.id_tipo_libro
    INTO v_id_tipo_libro
    FROM accounting.tipo_libro tl
    WHERE lower(tl.nombre) = CASE
        WHEN v_tipo_movimiento = 'Compra' THEN 'compras'
        ELSE 'ventas'
    END
      AND tl.activo = TRUE;

    IF v_id_tipo_libro IS NULL THEN
        RAISE EXCEPTION 'Tipo de libro no configurado para %', v_tipo_movimiento;
    END IF;

    v_fecha_inicio := to_date(p_periodo || '-01', 'YYYY-MM-DD');
    v_fecha_fin := (v_fecha_inicio + INTERVAL '1 month - 1 day')::DATE;

    SELECT
        COALESCE(sum(c.base_imponible + c.base_exonerada), 0),
        COALESCE(sum(c.igv), 0),
        COALESCE(sum(c.total), 0),
        count(*)
    INTO v_total_base, v_total_igv, v_total_general, v_comprobantes
    FROM accounting.comprobante c
    WHERE c.tipo_movimiento = v_tipo_movimiento
      AND c.estado = 'Registrado'
      AND c.fecha_emision BETWEEN v_fecha_inicio AND v_fecha_fin;

    IF v_comprobantes = 0 THEN
        RAISE EXCEPTION 'No existen comprobantes registrados para % en periodo %',
            v_tipo_movimiento,
            p_periodo;
    END IF;

    SELECT COALESCE(max(lc.version), 0) + 1
    INTO v_version
    FROM accounting.libro_contable lc
    WHERE lc.id_tipo_libro = v_id_tipo_libro
      AND lc.periodo = p_periodo;

    INSERT INTO accounting.libro_contable (
        id_tipo_libro,
        periodo,
        version,
        fecha_generacion,
        estado,
        total_base_imponible,
        total_igv,
        total_general
    )
    VALUES (
        v_id_tipo_libro,
        p_periodo,
        v_version,
        CURRENT_DATE,
        'Generado',
        v_total_base,
        v_total_igv,
        v_total_general
    )
    RETURNING id_libro INTO v_id_libro;

    INSERT INTO accounting.comprobante_libro (id_libro, id_comprobante)
    SELECT v_id_libro, c.id_comprobante
    FROM accounting.comprobante c
    WHERE c.tipo_movimiento = v_tipo_movimiento
      AND c.estado = 'Registrado'
      AND c.fecha_emision BETWEEN v_fecha_inicio AND v_fecha_fin
    ORDER BY c.fecha_emision, c.serie, c.numero;

    PERFORM audit.fn_registrar_evento(
        NULL,
        NULL,
        'AccountingSUNAT',
        'accounting.libro_contable',
        v_id_libro::VARCHAR,
        'GENERAR_LIBRO',
        'Exitoso',
        jsonb_build_object(
            'tipo', v_tipo_movimiento,
            'periodo', p_periodo,
            'version', v_version,
            'comprobantes', v_comprobantes,
            'total_base', v_total_base,
            'total_igv', v_total_igv,
            'total_general', v_total_general
        )
    );
END;
$$;

COMMIT;

# Database Schema Specification

Target database: PostgreSQL 16.

Target normal form: BCNF.

This document is a schema specification only. It does not provide executable SQL.

## Architecture Decisions Applied

| Decision | Schema Result |
|---|---|
| Role-based RBAC only for Sprint 1. | `USUARIO` and `ROL` are canonical; permission tables are deferred. |
| Add Departamento. | `DEPARTAMENTO` is required and referenced by `EMPLEADO`. |
| Add HorasExtra. | `HORAS_EXTRA` stores approved overtime by employee and period. |
| Add Compra/Venta discriminator. | `COMPROBANTE.tipo_movimiento` is required. |
| Add ledger entities. | `CUENTA_CONTABLE`, `PERIODO_CONTABLE`, `ASIENTO_CONTABLE`, and `DETALLE_ASIENTO` are canonical. |
| Persist/version tax configuration. | `CONFIG_TRIBUTARIA_VERSION` stores effective-dated tax rules. |
| Draft-only payroll recalculation. | `PLANILLA.estado` controls recalculation and finalization. |
| SUNAT book versioning bridge. | `COMPROBANTE_LIBRO` links vouchers to generated book versions. |

## Identity and Access

### ROL

| Attribute | Type | Rule |
|---|---|---|
| id_rol | INT | Primary key. |
| nombre | VARCHAR(30) | Unique. |
| descripcion | VARCHAR(200) | Optional. |
| nivel_acceso | INT | Required, positive. |
| activo | BOOLEAN | Default true. |

Expected roles: Administrador RRHH, Contador, Gerente Financiero, Administrador Sistema.

### USUARIO

| Attribute | Type | Rule |
|---|---|---|
| id_usuario | INT | Primary key. |
| username | VARCHAR(50) | Required, unique. |
| password_hash | VARCHAR(256) | Required, BCrypt hash. |
| nombre_completo | VARCHAR(200) | Required. |
| id_rol | INT | FK to `ROL`. |
| ultimo_acceso | TIMESTAMP | Optional. |
| activo | BOOLEAN | Default true. |
| fecha_creacion | TIMESTAMP | Default current timestamp. |

### TIPO_DESCUENTO

## Constraints

- `salario_base > 0`.
- `horas > 0`.
- `base_imponible >= 0`.
- `igv >= 0`.
- `total >= 0`.
- `periodo` and `PERIODO_CONTABLE.codigo` must follow `YYYY-MM`.
- `dni` must be 8 digits.
- `numero_documento` must be 11 digits when `tipo_documento = RUC`.
- `tipo_movimiento` must be `Compra` or `Venta`.
- `(serie, numero, id_tipo_comp, tipo_movimiento)` must be unique for vouchers.
- `(id_empleado, periodo)` must be unique for payroll.
- Existing payroll can be recalculated only when `PLANILLA.estado = Draft`.
- `LIBRO_CONTABLE(periodo, id_tipo_libro, version)` must be unique.
- `COMPROBANTE_LIBRO(id_libro, id_comprobante)` must be unique.
- Posted journal entries must have total debit equal to total credit.
- Journal entries cannot be posted to closed periods.

# Database Security and Operations Specification

## Security Principles

- Apply least privilege for application database roles.
- Protect database connections using SSL/TLS.
- Encrypt sensitive data at rest according to deployment capabilities.
- Never store plaintext passwords.
- Store user passwords as BCrypt hashes.

## PostgreSQL Roles

| Role | Purpose | Access |
|---|---|---|
| rol_app_read | Read-only application queries | SELECT on approved tables/views. |
| rol_app_write | Application writes | SELECT/INSERT/UPDATE on operational tables, no destructive admin rights. |
| rol_app_admin | Administrative operations | Controlled DDL/migration and maintenance rights. |

## Backup and Recovery

- Full backup: weekly.
- Incremental backup: daily.
- Restore procedure: documented and periodically tested.
- Backup scope: database schema, data, roles, procedures, configuration tables.

## Availability

- Target availability: 99.5% during working hours, Monday-Friday 08:00-18:00.
- Monitor slow queries using PostgreSQL facilities such as `pg_stat_statements`.
- Optimize indexes and plans for payroll, voucher, and book generation queries.

## Data Retention

- Store at least 5 years of payroll and voucher history.
- Design for future period-based partitioning and historical archiving.

## Audit

Each sensitive business operation must capture:

- User ID.
- Role.
- Date and time.
- Operation type.
- Affected entity.
- Result.
- Correlation ID where applicable.

# Accounting SUNAT Business Rules

## Voucher Rules

- Supported voucher types include Factura, Boleta, Nota de Credito, Nota de Debito, Guia de Remision, and Recibo por Honorarios.
- SUNAT codes from the PDF include: Factura `01`, Boleta `03`, Nota de Credito `07`, Nota de Debito `08`, Guia de Remision `09`.
- `tipo_movimiento` is required and must be `Compra` or `Venta`.
- RUC must have 11 digits when document type is RUC.
- DNI must have 8 digits when document type is DNI.
- A voucher with same type, movement, series, and number must not be duplicated.
- Valid states include Registrado, Validado, and Anulado.
- Only `Registrado` vouchers are included in generated books unless a later policy says otherwise.

## IGV Rules

- IGV rate must be resolved from `CONFIG_TRIBUTARIA_VERSION` using the voucher issue date.
- For taxable operations, `igv = base_imponible * (tasa_igv / 100)`.
- For exempt or unaffected operations, IGV is 0.
- Total is base plus IGV for taxable operations.
- Voucher must store the applied tax configuration version and rate used for historical traceability.

## SUNAT Book Rules

- Books are generated by period.
- Sales Book uses vouchers with `tipo_movimiento = Venta`.
- Purchase Book uses vouchers with `tipo_movimiento = Compra`.
- SUNAT row fields include period, document number/RUC, voucher type, series, number, date, taxable base, exempt base, IGV, and total.
- Books group vouchers by type and calculate totals by group and general totals.
- Generated books are stored in `LIBRO_CONTABLE`.
- Every generated book has a positive version number unique by period and book type.
- Vouchers are linked to generated books through `COMPROBANTE_LIBRO`.

## Configuration Rules

- Tax rates and SUNAT format metadata must be configurable.
- Format/rate changes must not require application code changes.
- Historical vouchers must retain the tax rate used at registration time.
- Tax configuration versions must not overlap for the same code and effective date range.

# Accounting SUNAT Database Specification

## Tables

- `COMPROBANTE`
- `TIPO_COMPROBANTE`
- `LIBRO_CONTABLE`
- `TIPO_LIBRO`
- `COMPROBANTE_LIBRO`
- `CONFIG_TRIBUTARIA_VERSION`

## Relationships

| Relationship | Cardinality |
|---|---|
| TIPO_COMPROBANTE to COMPROBANTE | 1:N |
| CONFIG_TRIBUTARIA_VERSION to COMPROBANTE | 1:N |
| TIPO_LIBRO to LIBRO_CONTABLE | 1:N |
| LIBRO_CONTABLE to COMPROBANTE_LIBRO | 1:N |
| COMPROBANTE to COMPROBANTE_LIBRO | 1:N |

## Constraints

- `COMPROBANTE.id_tipo_comp` is required.
- `COMPROBANTE.tipo_movimiento` is required and must be `Compra` or `Venta`.
- `COMPROBANTE.serie` and `COMPROBANTE.numero` are required.
- Voucher duplicate prevention must include type, movement, series, and number.
- `COMPROBANTE.fecha_emision` is required.
- `COMPROBANTE.tipo_documento` and `COMPROBANTE.numero_documento` are required.
- RUC document numbers must be 11 digits.
- DNI document numbers must be 8 digits.
- `COMPROBANTE.base_imponible >= 0`.
- `COMPROBANTE.igv >= 0`.
- `COMPROBANTE.total >= 0`.
- `COMPROBANTE.id_config_tributaria_version` references the applied tax configuration version.
- `TIPO_COMPROBANTE.codigo_sunat` is required.
- `LIBRO_CONTABLE.periodo` follows `YYYY-MM`.
- `LIBRO_CONTABLE.version` is unique by type and period.
- `COMPROBANTE_LIBRO(id_libro, id_comprobante)` is unique.
- Tax configuration versions for the same code must not overlap by effective date range.

## Indexes

- `COMPROBANTE(tipo_documento, numero_documento)`.
- `COMPROBANTE(fecha_emision)`.
- `COMPROBANTE(tipo_movimiento, fecha_emision)`.
- `COMPROBANTE(id_tipo_comp, fecha_emision)`.
- `COMPROBANTE(serie, numero, id_tipo_comp, tipo_movimiento)`.
- `LIBRO_CONTABLE(periodo, id_tipo_libro, version)`.
- `COMPROBANTE_LIBRO(id_libro)`.
- `COMPROBANTE_LIBRO(id_comprobante)`.
- `CONFIG_TRIBUTARIA_VERSION(codigo, version)`.

## Procedure Dependency

SUNAT book generation depends on `sp_generar_libro(tipo, periodo)` or an equivalent application service with the same transactional and output guarantees.

The procedure must create a new `LIBRO_CONTABLE.version` for the selected type/period and link included vouchers through `COMPROBANTE_LIBRO`.

# Authentication Business Rules

## Credential Rules

- Username is required.
- Username must have at least 3 alphanumeric characters.
- Password is required.
- Password is compared only through BCrypt hash verification.
- Disabled users cannot authenticate.

## Token Rules

- JWT expiration default is 1 hour, configurable by environment.
- JWT claims must include user ID, full name, role, and expiration.
- Expired tokens are invalid.
- Invalid tokens are rejected before the request reaches the protected use case.

## Failed Login Rules

- Failed attempts are counted per username.
- After 3 failed attempts, the account is blocked for 15 minutes.
- Error messages must not reveal whether username or password was incorrect.

## Authorization Rules

- The dashboard must be generated from role access rules.
- Users cannot see unauthorized module menu entries.
- Direct attempts to access unauthorized endpoints must be denied.
- Administrador Sistema has full module access.
- Sprint 1 uses role-based RBAC only. There is no permission catalog in the Sprint 1 domain or database model.
- Module access is derived directly from role: Administrador RRHH, Contador, Gerente Financiero, or Administrador Sistema.

## Session Rules

- Logout clears the in-memory token in the UI.
- Refresh tokens are out of Sprint 1 scope and are not part of the Sprint 1 API contract.
- Revoked tokens may be tracked in cache if Redis is used.

# General Ledger Business Rules

## Accounting Period Rules

- Accounting period code follows `YYYY-MM`.
- Journal entries can be created only in open periods.
- Closed periods cannot receive new posted entries.

## Account Rules

- Account code is unique.
- Account type must classify the account for reporting: Asset, Liability, Equity, Income, Cost, or Expense.
- Inactive accounts cannot be used in new journal entry lines.

## Journal Entry Rules

- A journal entry starts as `Draft`.
- A journal entry can be posted only when total debits equal total credits.
- Posted entries cannot be edited.
- Cancelled entries must preserve audit traceability.
- Journal entries can reference source module and source entity, such as Payroll or Accounting SUNAT.

## Reporting Rules

- Balance sheet uses posted ledger lines grouped by Asset, Liability, and Equity accounts.
- Income statement uses posted ledger lines grouped by Income, Cost, and Expense accounts.
- Draft and cancelled entries are excluded from financial reports.

# Payroll Business Rules

## Employee Validation Rules

- DNI is required, numeric, exactly 8 digits, and unique.
- Names and surnames are required, letters and spaces only, minimum 2 characters.
- Birth date is required and employee must be at least 18 years old.
- Hire date is required and cannot be future date.
- Salary base is required and must be greater than S/ 0.00.
- Department is required once the department catalog is enabled for payroll/report filtering.
- Discount type is required and must be AFP or ONP.
- Bank is required.
- Account number is required and must have 14 to 20 digits.

## Payroll Calculation Rules

- Payroll is calculated for a selected period in `YYYY-MM`.
- Only active employees are included.
- Gross salary starts from salary base.
- The PDF defines hourly rate as `salary_base / 240`.
- Overtime is stored in `HORAS_EXTRA` by employee and period.
- Overtime is based on hours worked and hourly rate; the PDF notes 25% surcharge for first two overtime hours and 35% for subsequent hours.
- AFP discount is 10% of gross salary.
- ONP discount is 13% of gross salary.
- Gratification is `gross_salary / 6`.
- CTS monthly estimate is `(gross_salary + gratification) / 12`.
- Total discounts include AFP or ONP and additional discounts when configured.
- Net pay is `gross_salary - total_discounts`.
- Payroll status starts as `Draft`.
- Existing payroll can be recalculated only while status is `Draft`.
- Payroll with status `Finalized` cannot be overwritten; it requires a cancellation or adjustment workflow outside Sprint 1.

## Persistence Rules

- Payroll results must be stored by employee and period.
- Payroll detail must store AFP, ONP, CTS, gratification, and additional discounts.
- Payroll detail must store overtime amount used by the calculation.
- Recalculation policy must block changes unless the existing payroll is still `Draft`.

## Export Rules

- Payslip PDF export is available only after payroll has been calculated.
- Payroll Excel export must include all visible result columns and totals.
- Exported amounts must match persisted payroll results.

# Payroll Database Specification

## Tables

- `EMPLEADO`
- `TIPO_DESCUENTO`
- `DEPARTAMENTO`
- `HORAS_EXTRA`
- `PLANILLA`
- `DETALLE_PLANILLA`

## Relationships

| Relationship | Cardinality |
|---|---|
| TIPO_DESCUENTO to EMPLEADO | 1:N |
| DEPARTAMENTO to EMPLEADO | 1:N |
| EMPLEADO to HORAS_EXTRA | 1:N |
| EMPLEADO to PLANILLA | 1:N |
| PLANILLA to DETALLE_PLANILLA | 1:N |

## Constraints

- `EMPLEADO.dni` unique.
- `EMPLEADO.salario_base > 0`.
- `EMPLEADO.fecha_ingreso <= current date`.
- `EMPLEADO.id_departamento` references `DEPARTAMENTO`.
- `TIPO_DESCUENTO.nombre` unique.
- `TIPO_DESCUENTO.porcentaje >= 0`.
- `DEPARTAMENTO.nombre` unique.
- `HORAS_EXTRA.periodo` follows `YYYY-MM`.
- `HORAS_EXTRA(id_empleado, periodo)` is unique.
- Overtime hours cannot be negative.
- `PLANILLA.periodo` follows `YYYY-MM`.
- `PLANILLA.estado` is one of `Draft`, `Calculated`, `Finalized`, `Cancelled`.
- Existing `PLANILLA` rows can be recalculated only when `estado = Draft`.
- `PLANILLA.total_neto = total_bruto - total_descuentos` as an invariant.
- `DETALLE_PLANILLA` monetary fields default to 0.

## Indexes

- `EMPLEADO(dni)`.
- `EMPLEADO(activo)`.
- `EMPLEADO(id_departamento)`.
- `EMPLEADO(id_tipo_descuento)`.
- `HORAS_EXTRA(id_empleado, periodo)`.
- `PLANILLA(periodo)`.
- `PLANILLA(id_empleado, periodo)`.
- `DETALLE_PLANILLA(id_planilla)`.

## Procedure Dependency

Payroll calculation depends on `sp_calcular_planilla(periodo)`.

The procedure must execute atomically and return structured calculation results for the UI and API.

# Reports Business Rules

## Dashboard Rules

- Dashboard values are calculated for the selected period.
- Default period is the current month.
- KPIs must refresh when the period changes.
- Users only see report modules permitted by role.

## Balance Sheet Rules

- Balance sheet groups data into assets, liabilities, and equity.
- Values are calculated for a selected date or period end.
- The report must expose enough detail to support strategic decision-making.
- Balance sheet values must be derived from posted `DETALLE_ASIENTO` lines grouped through `CUENTA_CONTABLE`.
- Draft journal entries are excluded.

## Income Statement Rules

- Income statement groups income, costs, expenses, and net result.
- Net utility is income minus costs and expenses.
- Margin is derived from net utility and income when income is greater than zero.
- Income statement values must be derived from posted `DETALLE_ASIENTO` lines grouped through `CUENTA_CONTABLE`.

## Consolidated Payroll Report Rules

- Payroll report totals must match persisted payroll records.
- Report includes gross pay, discounts, CTS, gratification, and net pay.
- Period filters must be applied consistently across all totals.
- Department filters use `DEPARTAMENTO` assigned to employees.

## Export Rules

- PDF export is intended for distribution.
- Excel export is intended for analysis.
- Exported report values must match the UI view and selected filters.
