# Accounting SUNAT Business Rules

## Voucher Rules

- Supported voucher types include Factura, Boleta, Nota de Credito, Nota de Debito, Guia de Remision, and Recibo por Honorarios.
- SUNAT codes from the PDF include: Factura `01`, Boleta `03`, Nota de Credito `07`, Nota de Debito `08`, Guia de Remision `09`.
- `tipo_movimiento` is required and must be `Compra` or `Venta`.
- RUC must have 11 digits when document type is RUC.
- DNI must have 8 digits when document type is DNI.
- A voucher with same type, movement, series, and number must not be duplicated.
- SUNAT codes `07` (Nota de Credito) and `08` (Nota de Debito) require a `comprobanteReferenciaId` pointing to a Registrado voucher with the same Compra/Venta movement. Other voucher types cannot carry that reference.
- A Nota de Credito reduces the referenced transaction using the exact inverse debit/credit orientation for its own positive amounts. A Nota de Debito increases it and uses the ordinary Compra/Venta orientation. The original voucher and entry remain unchanged.
- At least one of taxable or exempt base must be greater than zero; a zero-total voucher cannot produce the required source journal entry.
- Persisted states are `Registrado` and `Anulado` only.
- SUNAT validation is computed and does not change voucher state.
- Only `Registrado` vouchers are eligible for new book versions.
- A voucher linked to a generated version cannot change business fields; it may be annulled. Historical bridge snapshots remain unchanged.

## IGV Rules

- IGV rate must be resolved from `CONFIG_TRIBUTARIA_VERSION` using the voucher issue date.
- For taxable operations, `igv = base_imponible * (tasa_igv / 100)`.
- For exempt or unaffected operations, IGV is 0.
- Total is base plus IGV for taxable operations.
- Voucher must store the applied tax configuration version and rate used for historical traceability.

## SUNAT Book Rules

- Books are generated by period.
- Sales Book uses vouchers with `tipo_movimiento = Venta`.
- Purchase Book uses vouchers with `tipo_movimiento = Compra`.
- SUNAT row fields include period, document number/RUC, voucher type, series, number, date, taxable base, exempt base, IGV, and total.
- Books group vouchers by type and calculate totals by group and general totals.
- Generated books are stored in `LIBRO_CONTABLE`.
- Every generated book has a positive version number unique by period and book type.
- Vouchers are linked to generated books through `COMPROBANTE_LIBRO`.
- Validation returns `Valida`, `ConObservaciones`, or `Bloqueada`; only the first two allow generation and warnings must be acknowledged.
- Generation is a command and always creates the next version; retrieval never creates data.
- A generated book stores the effective `CONFIG_SUNAT_FORMATO` and authenticated generator.
- Each `COMPROBANTE_LIBRO` row stores snapshot amounts and the voucher's applied `CONFIG_TRIBUTARIA_VERSION`.
- Book generation locks the eligible Registrado voucher rows, reruns validation, and uses that same locked set for header totals and immutable bridge snapshots.

## Configuration Rules

- Tax rates and SUNAT format metadata must be configurable.
- Format/rate changes must not require application code changes.
- Historical vouchers must retain the tax rate used at registration time.
- Tax configuration versions must not overlap for the same code and effective date range.
- SUNAT format versions follow Draft -> Active -> Closed, cannot overlap by book type, and Active/Closed versions are immutable.
- `estructura_json` is `{ "columns": [...] }` with exactly the eleven unique canonical tokens: periodo, tipoDocumento, numeroDocumento, tipoComprobante, serie, numero, fechaEmision, baseImponible, baseExonerada, igv, total. Array order is export order; each token maps directly to the same-named immutable bridge snapshot field. Unknown/missing/duplicate tokens are invalid.
- Direct electronic submission to SUNAT is out of scope.

## Out of Scope

The following SUNAT tax regimes are explicitly out of scope for the MVP basic ERP and must not be implemented. They are documented here to prevent ambiguity during sprints.

- **Detracciones**: the SUNAT advance-payment withholding system for purchases of goods, services, and construction. No `detraccion` fields, no `SP_DETRACCION` calculation, and no detraction voucher rows are part of the MVP.
- **Retenciones de Renta de cuarta categoria**: the withholding applied to `Recibo por Honorarios` (voucher type `02`). `Recibo por Honorarios` remains a supported voucher type for registration and book inclusion, but the withholding computation, retention voucher, and resulting renta retention accounting entries are not implemented.
- **Impuesto a la Renta de quinta categoria**: monthly IR withholding on salaries. See `specs/payroll/business-rules.md` for the payroll-side deferral. No IR-5ta accounting entries are generated.
- **Percepciones**: the advance-payment collection system on sales. Not implemented.
- **Direct PLE submission to SUNAT**: book generation produces immutable versioned rows suitable for export; the actual SUNAT electronic submission channel is out of scope.
- **Retenciones del regimen MYPE**: not applicable to the MVP scope.

These exclusions are governed by `docs/p0-decisions.md` scope boundaries. Adding any of them later requires a new spec under `specs/accounting-sunat/` and an explicit scope decision.
